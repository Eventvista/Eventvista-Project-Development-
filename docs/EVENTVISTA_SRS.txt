1. INTRODUCTION
1.1 Purpose
This Software Requirements Specification (SRS) document defines the comprehensive functional,
non-functional, business, and architectural requirements for Eventvista. Eventvista is an AI-driven,
end-to-end event-organisation ecosystem built to democratise professional-grade spatial planning
and resource forecasting for everyday users.
1.2 Problem Statement & Specific Objectives
Traditional event coordination remains fragmented, expensive, and manual. Casual event planners
struggle to balance structural room dimensions, local budget ceilings, and logistical safety
calculations. Existing software options (e.g., Cvent, Social Tables) cater exclusively to enterprise
venue operators, demanding complex CAD layout uploads or extensive pre-rendered design
matrices.
Strategic Project Objectives:
To build an integrated web platform that reduces event setup times by over 70% for everyday users.
To provide a high-fidelity image-to-3D transformation tool capable of converting real-world
photos into functional layouts within 15 seconds.
To embed predictive analytical tools that continuously manage the project's "Triple Constraints"
(Scope, Time, and Budget).
To create a secure marketplace architecture for instant local component procurement.

1.3 Scope
Eventvista eliminates the fragmentation of traditional event planning by combining administrative
event workflows with an advanced generative AI spatial model pipeline. The system takes a
standard 2D image of an empty venue captured by a user's mobile device, interprets the room
boundaries, maps architectural constraints, processes event metrics (guest numbers, resource
availability, budget limits), and synthesises an interactive, customisable 3D event layout canvas.
1.4 Value Proposition
'Eventvista's core value rests on three distinct pillars:
Democratised Spatial Planning: Eliminates spatial layout guesswork. A user can step inside an
empty hall, capture a panoramic image, and instantly review how 200 dining chairs, a banquet table
array, and a performance stage fit dynamically into the space.
Real-Time Predictive Budget Optimisation: The Groq-powered backend acts as a virtual project
consultant, warning users if adding a furniture element violates structural boundaries, safety
clearances, or allocated budget metrics.
Frictionless Transaction Execution: Integrating the Ms-Pesa API means African event organisers
can coordinate vendor execution directly within localised financial grid.

2. GENERAL DESCRIPTION
2.1 Product Perspective
Eventvista operates as a cross-platform web application optimised for desktop customisation and
mobile image capturing. The system relies heavily on external high-performance APIs for lowlatency computation: the Groq API handles deterministic configuration logic, predictive resource
mapping, and user parameter interpretation, while the open-source TRELLIS model pipeline,
hosted on Hugging Face, parses visual structural boundaries to deliver interactive 3D assets to the
user's browser.

2.2 Product Functions
Automated venue parsing via image upload.
Predictive resource, asset, and budget allocation modelling.
Interactive, browser-based drag-and-drop 3D space modification.
End-to-end event management metrics tracking (guest registration, automated messaging
pipelines).
Localised payment orchestration.

2.3 User Characteristics
The target user is an everyday consumer or casual corporate planner with zero technical training in
Computer-Aided Design (CAD), spatial engineering, or advanced spreadsheet accounting. The
interface prioritises high-accessibility mechanics over clinical design configurations.

2.4 Constraints
This section outlines the technical limitations and boundaries that must be considered during the
development and operation of the Eventvista platform. These constraints ensure that the system
remains stable and predictable for all users:
Hardware Limitations: Eventvista requires a device with a modern Graphics Processing Unit
(GPU) to handle real-time 3D rendering. Users on older hardware or devices without dedicated
graphics acceleration may experience reduced performance, such as slower 3D model generation
or lower frame rates within the venue designer.
Browser Compatibility: The platform relies on WebGL and React Three Fibre to display 3D
environments directly in the web browser. Therefore, users must have a modern, up-to-date web
browser (such as Chrome, Edge, Safari, or Firefox) that supports WebGL 2.0. Older browsers or
those with strict security settings that block WebGL will be unable to render the 3D canvases.
API Rate Limits: To maintain performance and manage operational costs, Eventvista operates
under specific API rate limits. This includes caps on the number of requests sent to the Groq AI
engine and the Hugging Face model per user session. Exceeding these limits during a single session
will result in temporary service throttling, requiring the user to wait before performing additional
AI-driven tasks.

2.5 Assumptions and Dependencies
This section identifies the critical external factors and services that Eventvista relies on to function
correctly. The successful operation of the platform is dependent on the availability and
performance of these third-party systems:
Dependency on Hugging Face GPU Availability: The ability to convert 2D venue images into 3D
models relies on the TRELLIS AI model hosted on Hugging Face. Eventvista assumes that these
hosted GPU spaces remain available. If the Hugging Face service encounters downtime or high
server traffic, the 3D generation feature may experience delays or become temporarily unavailable.
Groq API Response Latency: 'Eventvista's resource forecasting and budgeting features are
powered by the Groq LPU engine. We assume that the Groq API will maintain its standard lowlatency performance. Because the system is designed for real-time interaction, any significant
spikes in network latency or service degradation from the Groq API provider will directly affect the
speed at which users receive budget calculations and planning insights.
Reliable Cloud Infrastructure: It is assumed that our hosting partners, Vercel and Render, will
maintain 99.9% uptime. The application depends on these services for constant access to the
database and frontend assets; therefore, any unplanned outages in these hosting environments
would impact the overall accessibility of the Eventvista platform.

3.4 Core Technology Stack Details
User UI Layer: Next.js framework compiled with Tailwind CSS, deployed on Vercel for global
edge network delivery, fast continuous integration (CI/CD), and optimal React Three Fiber (R3F)
rendering.
Core Logic Layer: Node.js backend managing an Express.js router structure, deployed on Render
as a Web Service to ensure seamless API connections and straightforward GitHub deployment
hooks.
Database Grid: MongoDB for flexible schema modelling of variable spatial floor arrangements,
combined with PostgreSQL to secure financial processing, user metadata registries, and
transaction accounts.
Computational Framework Interfaces: Groq API: Interfaced using the ultra-low latency `llama3.1-70b-versatile` system model to structure spatial configuration parameters instantly.
TRELLIS 3D Pipeline: Open-source generative framework (`TRELLIS.2-4B`) operating through
isolated GPU endpoints on Hugging Face Spaces to generate highly accurate GLTF spatial meshes
from 2D venue images.
Payment Orchestration Network: Local mobile-first transaction clearing managed natively
through the Mpesa API Gateway.
Document Generation & Approval Stack:
MeshLab: An open-source backend tool programmed to automatically convert the raw 3D venue
models generated by the AI into the U3D format, which is required for PDF embedding.
LaTeX: An open-source typesetting system that will pull event specifications (such as audience
size, budget, and venue details) from the database, format them into a professional document, and
embed the interactive U3D model directly into the centre of the page.
OpenSign: A free, open-source digital signature platform. Once the PDF is generated, OpenSign
will securely email it to the client and provide a legally binding electronic signature slot for final
approval.

4.2 Hardware Interfaces
To ensure a seamless user experience, particularly when interacting with the 3D venue designer,
the following hardware requirements apply. These ensure that WebGL and React Three Fiber
canvases render with sufficient stability and frame rates:
Graphics Processing Unit (GPU): A dedicated or integrated GPU capable of supporting WebGL
2.0 is required. Modern hardware, such as that found in standard laptops or tablets released within
the last 5 years, is generally sufficient for real-time 3D rendering.
Memory (RAM): A minimum of 8GB of system RAM is recommended to support smooth browser
performance while maintaining active AI model processes and 3D asset manipulation.
Display Resolution: A minimum screen resolution of 1366x768 pixels is required to interact
effectively with the drag-and-drop toolbars and canvas controls without user interface overlap.
Input Devices: A pointer device (mouse or trackpad) is highly recommended over touch-only
interfaces, as the 3D venue design tool requires precise clicking, dragging, and rotation control for
spatial planning.

4.3 Software Interfaces
Eventvista utilises a modern decoupled software architecture to ensure scalability and ease of
maintenance. The interfaces between the frontend and backend are managed as follows:
Frontend (Next.js): The user interface is built using Next.js. It interacts with the backend solely
through defined API endpoints. The frontend is responsible for state management, user
authentication flows, and rendering the 3D interactive components.
Backend (Node.js/Express): The backend serves as the core API gateway. It exposes RESTful API
routes that the frontend calls to perform CRUD (Create, Read, Update, Delete) operations. This
includes retrieving event data, processing budget calculations, and saving 3D layout
configurations.

Data Exchange Format: All communication between the frontend and backend is conducted using
standard JSON (JavaScript Object Notation), ensuring consistent data parsing and structure across
the entire application stack.

4.4 Communications Interfaces
To maintain high standards of security and real-time interaction, Eventvista employs the following
communication protocols:
Secure Transport (HTTPS): All data transmitted between the client browser and the server is
encrypted using Transport Layer Security (TLS/HTTPS). This protects sensitive information, such
as payment details and user credentials, from interception during transmission.
Real-time Communication (WebSockets): The messaging and collaboration hub uses WebSockets
to establish a persistent, bidirectional communication channel between the user and the platform.
This allows for instantaneous updates, such as incoming messages or notification alerts, without
requiring the user to refresh the page.
RESTful Protocol: The primary interface for general application logic follows RESTful design
principles. This uses standard HTTP methods (GET, POST, PUT, DELETE) to interact with
resources, ensuring a clean, predictable, and stateless interface between the client and the server.

6. NON-FUNCTIONAL REQUIREMENTS

NFR-2.1: Financial processing channels must integrate absolute cryptographic protocols matching

NFR-2.2: User database structures must obfuscate passwords using industry-standard hashing

6.3 Scalability and Availability
NFR-3.1: The frontend architecture deployed on 'Vercel's Edge Network must automatically cache
dynamic routes and autoscale to handle high-traffic surges during peak event registration times.
NFR-3.2: The backend logic managed on Render must guarantee continuous deployment from the
main Git repository and maintain a minimum core operational availability benchmark of 99.9%
annually.

7. Project Management & Analysis Models

7.1 Project Implementation Plan & Methodology (Agile Framework)
For the Eventvista project, we utilise an Agile Framework. Unlike traditional linear (Waterfall)
models where you finish one stage before starting the next, Agile is iterative and flexible. It allows
the team to develop the software in small, manageable cycles (often called "sprints").
Why we chose Agile for Eventvista:
Flexibility: As we develop complex AI-driven spatial planning tools, requirements may evolve
based on user testing. Agile allows us to adapt to these changes without derailing the entire project.
Continuous Feedback: We involve stakeholders and users throughout the process, ensuring the
final output actually solves the problem of inefficient event planning.
Incremental Delivery: By breaking the 12 phases into smaller chunks, we can test and refine
features (like the 3D venue designer) early, rather than waiting until the end to discover issues.

8. LEGAL AND ETHICAL CONSIDERATIONS
Data Protection Compliance: Absolute structural alignment with international standards and the
Kenya Data Protection Act of 2019. Venue spatial pictures and user contact metrics will be strictly
isolated and encrypted using TLS 1.3.
Ethical Safety Guards: Spatial generation scripts will incorporate automatic alerts preventing
modifications that knowingly compromise regional safety rules, indoor escape route corridors, or
legal fire capacity benchmarks.
8.2 Ethical and Explainable Use of AI
Eventvista relies heavily on Artificial Intelligence to convert 2D images into 3D models and to
automate budget forecasting. It is a core ethical requirement that this AI operates transparently.
"Explainable AI" means that the system does not act as a mysterious 'black box'. When Eventvista's
AI restricts a user from placing too many tables in a 3D canvas, or when it calculates a specific
budget estimate, it will provide clear, readable prompts explaining why (e.g., "This layout exceeds
the safe fire-exit capacity for this room size"). Furthermore, our AI models are programmed to
remain neutral and unbiased, ensuring that all venue designs and vendor recommendations are
generated fairly and without prejudice.
8.3 Oversight Bodies
To ensure compliance with legal and ethical standards, Eventvista's development and operations
adhere to the guidelines set by relevant oversight authorities. As the platform handles personal user
information and M-Pesa payment details, it must align with the regulations of the Office of the
Data Protection Commissioner (ODPC) in Kenya. Additionally, because this is a capstone project
developed at The Technical University of Kenya, the system's design and research methodology
are subject to the oversight of the 'university's Academic and Ethical Review Boards. This ensures
the project maintains high academic integrity and responsible software engineering practices.

9.2 Future Iterations and Recommendations
While the current version of Eventvista successfully delivers an interactive 3D venue designer and
automated budgeting, there is significant room for future growth. For future iterations, we
recommend the following enhancements:
Virtual Reality (VR) Integration: Allowing event organisers and clients to put on a VR headset and
physically "walk through" the AI-generated 3D venue before making final decisions.
Live Vendor Bidding System: Upgrading the vendor module so that local caterers, DJs, and
decorators can view a user's 3D event canvas and submit competitive price quotes directly through
the platform.
Paperless Vendor Contracting: Expanding the OpenSign integration beyond client approvals so
that when local vendors (such as caterers or DJs) win a bid on the Live Vendor Bidding System, the
system automatically generates and sends them a secure, legally binding digital contract to sign.
Real-time AI Decor Suggestions: Training the AI to automatically suggest colour schemes and
decor placements based on the specific cultural or thematic requirements of the event.

9.3 Project Potential for Research and Teaching
Eventvista serves as a highly practical case study for academic institutions. Within the fields of
Information Science, Software Engineering, and Infopreneurship, this project demonstrates
exactly how to integrate modern AI APIs (such as Groq and Hugging Face) with traditional web
development to solve real-world business problems. The 'project's architecture, system logic, and
complete source code can be utilised as a teaching tool by university lecturers. It provides future
students with a tangible example of how to manage a full Software Development Life Cycle
(SDLC) while building commercially viable, AI-driven software.
